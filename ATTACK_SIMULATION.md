# Lighthouse — Live Attack Simulation Guide

Real traffic is generated by Docker containers, captured by Suricata,
and classified by the CIC 2017 SMOTE model in real time.

```
attacker-hping ──┐
attacker-msf   ──┤──▶  victim (nginx)  ◀── Suricata (eve.json)
                 │                              │
                 └──────────────────────────────┘
                                                │
                                        suricata_bridge.py
                                        (19 CIC features)
                                                │
                                        ML model (XGBoost + LightGBM)
                                                │
                                        live_detector.py
                                        (terminal + logs/detections.json)
```

---

## 1. Start the full stack

```bash
# From the repo root
docker compose -f infra/docker-compose.attack.yml up -d

# Watch the detector output in real time
docker logs -f lh-detector
```

Wait ~15 seconds for Suricata to start sniffing before launching attacks.

---

## 2. Launch attacks

### hping3 attacks (DDoS, DoS, PortScan, Brute Force)

```bash
# Copy the attack script into the container (first time only)
docker cp infra/attacker/hping_attacks.sh lh-attacker-hping:/home/attacks.sh

# DDoS — SYN flood on port 80
docker exec lh-attacker-hping sh /home/attacks.sh ddos 172.28.0.10

# DDoS HTTPS — SYN flood on port 443
docker exec lh-attacker-hping sh /home/attacks.sh ddos-https 172.28.0.10

# DoS Hulk — HTTP ACK/SYN flood
docker exec lh-attacker-hping sh /home/attacks.sh dos-hulk 172.28.0.10

# DoS Slowloris — many half-open connections
docker exec lh-attacker-hping sh /home/attacks.sh dos-slow 172.28.0.10

# Port scan — all ports 1-1024
docker exec lh-attacker-hping sh /home/attacks.sh portscan 172.28.0.10

# Brute force SSH
docker exec lh-attacker-hping sh /home/attacks.sh bruteforce 172.28.0.10

# Run ALL hping attacks in sequence
docker exec lh-attacker-hping sh /home/attacks.sh all 172.28.0.10
```

### Metasploit attacks (Heartbleed, Web Attack, Brute Force)

```bash
# Load all attack modules from the RC file
docker exec -it lh-attacker-msf msfconsole -r /home/msf/attacks.rc

# Or run a single module interactively
docker exec -it lh-attacker-msf msfconsole -q -x \
  "use auxiliary/scanner/ssl/openssl_heartbleed; \
   set RHOSTS 172.28.0.10; set RPORT 443; run; exit"
```

## 3. Watch detections

Terminal (colour-coded):
```bash
docker logs -f lh-detector
```

Example output:
```
!! DDoS            172.28.0.20     -> 172.28.0.10:80   TCP/?   dur=2.1ms   prob=99%
!! PortScan        172.28.0.20     -> 172.28.0.10:22   TCP/?   dur=0.3ms   prob=87%
!! Heartbleed      172.28.0.21     -> 172.28.0.10:443  TCP/tls dur=890.0ms prob=95%
   BENIGN          192.168.1.5     -> 172.28.0.10:80   TCP/http dur=45.2ms prob=3%
```

JSON log file (inside detector container):
```bash
docker exec lh-detector tail -f /app/logs/detections.json | python -m json.tool
```

Copy logs to host:
```bash
docker cp lh-detector:/app/logs/detections.json ./detections.json
```

## Architecture notes

| Component | What it does |
|---|---|
| `lh-victim` | nginx on 172.28.0.10 — the attack target |
| `lh-suricata` | Sniffs `docker0` bridge, writes `eve.json` with flow records |
| `lh-detector` | Reads `eve.json`, extracts 19 CIC features, runs ML model |
| `lh-attacker-hping` | hping3 for SYN floods, scans, brute force |
| `lh-attacker-msf` | Metasploit for Heartbleed, Web Attacks, exploit modules |

**Why Suricata?**
The ML model was trained on CIC 2017 network flow features (packet counts,
byte totals, inter-arrival times, TCP flags). Suricata extracts these same
statistics from live traffic and writes them to `eve.json` — bridging the
gap between real packets and the model's expected input.
